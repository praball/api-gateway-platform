1. Why call User Service instead of userdb while validating userid for creating an order in Order service?

Because each microservice owns its own database.

User Service  <--> userdb
Order Service <--> orderdb

Order Service should never access userdb directly. Otherwise, services become tightly coupled, defeating the purpose of microservices.

--------x--------

2. Why not map Order and User with JPA?

That works in a monolith, where both entities share one database.

In microservices:

orderdb     |     userdb
 Order      |      User

They're in different databases (and often different servers), so @ManyToOne / @JoinColumn is not appropriate.

Instead, Order stores only:

Long userId;

and validates it by calling User Service.

--------x--------

3. Why not use FeignClient to call User Service?

FeignClient is a declarative web service client that makes writing web service clients easier. However, it has some limitations:
- It can be hard to debug, as it hides the underlying HTTP calls.
- It can introduce tight coupling between services, as it relies on the service's interface.
- It can be less flexible, as it may not support all HTTP features or customizations.

In this project, we use RestTemplate instead of FeignClient for the following reasons:
- RestTemplate provides more control over the HTTP requests and responses, allowing for better error handling and logging.
- RestTemplate is more flexible, as it can be easily customized to support different HTTP features and configurations.
- RestTemplate is more suitable for microservices architecture, as it allows for better separation of concerns and reduces tight coupling between services.

4. Doesn't the use of .block() defeat the purpose of using WebFlux?

Yes, but it's okay here. Your Order Service is using:
spring-boot-starter-web

which is Spring MVC (blocking). So blocking is normal. If the entire service were built on WebFlux, then we'd return Mono<Order> and avoid .block().

5. Why to use Redis for rate limiting instead of in-memory?

Suppose you have 3 gateway instances.

     Gateway-1        |       Gateway-2         |       Gateway-3

If each keeps its own counter:

Gateway1 : 5 requests |  Gateway2 : 5 requests  |  Gateway3 : 5 requests

The client effectively gets 15 requests instead of 5.

Redis solves this.

Gateway1 ─┐
Gateway2 ─┼── Redis Counter = 5
Gateway3 ─┘

All gateway instances share the same counter.

That's why it's called distributed rate limiting.

6. CompletableFuture vs Mono

CompletableFuture:
- One async result.
- Java concurrency API.
- Doesn't model streams or backpressure.

Mono
- One async result (0 or 1 value).
- Part of Reactor/WebFlux.
- Integrates with Spring WebFlux and supports reactive pipelines.

Think:

CompletableFuture ≈ async task
Mono ≈ async task + reactive pipeline
